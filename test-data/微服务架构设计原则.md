# 微服务架构设计原则

> 本文档总结了微服务架构的核心设计原则和最佳实践

---

## 一、什么是微服务

微服务架构是一种将单个应用程序开发为一组小型服务的方法，每个服务：
- 运行在自己的进程中
- 通过轻量级机制（通常是 HTTP API）通信
- 围绕业务能力构建
- 可独立部署

## 二、核心设计原则

### 2.1 单一职责原则

每个微服务应该只有一个改变的理由。服务边界应该基于：
- **业务领域**（如订单、支付、用户）
- **数据一致性需求**
- **扩展性需求**

### 2.2 去中心化治理

- 不同的服务可以使用不同的技术栈
- 没有"一刀切"的架构
- 团队自主选择最适合的工具

### 2.3 去中心化数据管理

- 每个服务拥有自己的数据库（Database per Service）
- 避免共享数据库
- 使用事件驱动架构实现数据最终一致性

### 2.4 容错设计

- **熔断器模式**：失败的服务快速失败，不拖垮整个系统
- **降级策略**：核心功能优先，非核心功能可降级
- **超时控制**：所有外部调用必须设置超时

### 2.5 自动化运维

- 自动化部署（CI/CD）
- 自动化监控和告警
- 自动化扩缩容

## 三、服务拆分策略

### 3.1 按业务领域拆分

```
电商系统:
├── 用户服务 (User Service)
├── 商品服务 (Product Service)
├── 订单服务 (Order Service)
├── 支付服务 (Payment Service)
├── 库存服务 (Inventory Service)
└── 物流服务 (Shipping Service)
```

### 3.2 按变更频率拆分

- 频繁变更的功能独立成服务
- 稳定的功能可以合并

### 3.3 避免的拆分方式

- ❌ 按 CRUD 操作拆分（UserService, OrderService...）
- ❌ 按技术层拆分（Web Service, DB Service）
- ❌ 过度拆分（一个函数一个服务）

## 四、服务间通信

### 4.1 同步通信

| 协议 | 适用场景 | 注意事项 |
|------|---------|---------|
| HTTP/REST | 通用 API 调用 | 设置超时，实现重试 |
| gRPC | 高性能内部调用 | 需要 HTTP/2 支持 |
| GraphQL | 灵活查询场景 | 注意 N+1 问题 |

### 4.2 异步通信

| 协议 | 适用场景 |
|------|---------|
| RabbitMQ | 任务队列，事件分发 |
| Kafka | 事件溯源，流处理 |
| Redis Pub/Sub | 简单消息广播 |

## 五、数据一致性

### 5.1 最终一致性模式

微服务架构中，强一致性往往不可行，推荐最终一致性：

```
订单创建 → 发送事件 → 库存服务消费 → 扣减库存
         ↓
      支付服务消费 → 创建支付记录
```

### 5.2 Saga 模式

长事务拆分为一系列本地事务：

```
创建订单
  ↓
预扣库存 → 失败则补偿
  ↓
创建支付 → 失败则补偿（恢复库存）
  ↓
确认订单 → 失败则补偿
```

## 六、服务发现

### 6.1 客户端发现

- 服务注册到注册中心
- 客户端查询注册中心获取服务列表
- 客户端实现负载均衡

### 6.2 服务端发现

- 通过负载均衡器访问服务
- 负载均衡器查询注册中心
- 服务对客户端透明

### 6.3 常见方案

| 方案 | 特点 |
|------|------|
| Consul | HashiCorp 出品，功能完善 |
| etcd | CoreOS 出品，Kubernetes 默认 |
| Eureka | Netflix 出品，Java 生态常用 |
| Nacos | 阿里出品，支持配置管理 |

## 七、配置管理

### 7.1 配置外置

- 配置与代码分离
- 支持动态刷新
- 按环境隔离

### 7.2 常见方案

- Spring Cloud Config
- Consul KV
- etcd
- Apollo（携程开源）

## 八、监控与可观测性

### 8.1 三大支柱

1. **Metrics（指标）**：Prometheus + Grafana
2. **Logging（日志）**：ELK / Loki
3. **Tracing（链路追踪）**：Jaeger / Zipkin

### 8.2 关键指标

- 请求量（QPS）
- 响应时间（P50/P95/P99）
- 错误率
- 饱和度（CPU/内存使用率）

## 九、安全设计

### 9.1 认证授权

- OAuth2 / OIDC 统一认证
- JWT 传递用户信息
- RBAC 权限控制

### 9.2 服务间安全

- mTLS 双向认证
- 服务网格（Istio）
- API Gateway 统一入口

## 十、部署架构

### 10.1 容器化部署

```yaml
# Docker Compose 示例
services:
  user-service:
    build: ./user-service
    ports:
      - "8081:8080"
    environment:
      - DB_HOST=postgres
      - REDIS_HOST=redis

  order-service:
    build: ./order-service
    ports:
      - "8082:8080"
```

### 10.2 Kubernetes 部署

- Deployment 定义服务
- Service 服务发现
- Ingress 外部访问
- ConfigMap 配置管理
