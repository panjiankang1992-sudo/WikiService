## 一、项目概述

**目标**：打造一个完全离线的、面向团队的内部知识库系统，支持多源文档导入、语义搜索、知识图谱可视化。

**最终成果**：WikiService LightRAG 单机知识库，6 天内完成从需求分析到可部署的完整交付。

## 二、过程回顾

### 2.1 准备阶段：需求分析与方案选型（5/10）

**工具链**：
- 使用 **ima** 进行需求分析，梳理出团队知识库的四大核心需求：知识沉淀、全文检索、图谱关联、离线可用
- 通过 ima 调研开源方案，对比了 WeKnora、Danswer、FastGPT、Quivr 等候选项目

**选型决策**：

| 候选方案 | 优点 | 缺点 | 结论 |
|----------|------|------|------|
| WeKnora 全家桶 | 功能完整（Neo4j + pgvector + MCP） | 依赖 Docker 多容器，离线部署复杂 | 先尝试，后弃用 |
| FastGPT | 知识库 + 对话 | 强依赖 LLM API，无法离线 | 不适合 |
| **LightRAG 单机** | 轻量、纯 Python、JSON 存储 | 功能需自建 | **最终采用** |

**初始方案**：基于 WeKnora Docker Compose 的完整部署（Neo4j 图数据库 + PostgreSQL pgvector 向量库 + Crawl4AI 爬虫 + MCP Server + Prometheus 监控）。

这个阶段我们产出了详细的《WikiServer 服务器版 Wiki 完整实现方案》，涵盖了 15 天分阶段实施计划和风险缓解策略。

### 2.2 第一次转向：从 WeKnora 到 LightRAG（5/12）

实施 WeKnora 方案时发现了几个问题：

1. **Docker 复杂度高**：WeKnora 需要 Neo4j、PostgreSQL、Redis 等多个服务，离线环境部署繁琐
2. **运维成本大**：多容器编排、数据备份、监控告警 —— 团队规模不匹配
3. **过度设计**：MCP Server、Prometheus 监控对当前阶段是过早优化

**决策**：放弃 WeKnora 全家桶，转向 LightRAG 单机方案。

这一决策的核心判断是：**先解决"有没有"，再解决"好不好"**。一个能立刻跑起来的单机知识库，比一个架构完美但部署复杂的全家桶更有价值。

### 2.3 第二个关键调整：从"问答"到"知识查询"（5/14 → 5/16）

这是项目中最重要的一次方向校正。

**初始设计**：智能问答系统 —— 用户提问 → LLM 理解意图 → 检索文档 → LLM 生成答案。这是典型的 RAG 模式，类似 ChatGPT + 知识库。

**问题暴露**：
- 5/14：最初引入 LLM（DeepSeek Chat）做文档评分和答案生成，响应延迟数百毫秒
- 5/15 上午：LLM 评分结果不稳定，同一查询多次结果不一致
- 5/15 中午：决定移除 LLM 评分，改用纯关键词匹配，但"问答"的思维惯性仍在
- 用户反馈："智能问答查询出的材料没有给出推荐的理由，不要出现重复的文档"

**重新思考**：

我们做了一个重要的认知转换：用户需要的不是"AI 替你回答"，而是"AI 帮你找到最相关的文档"。这类似于 Google 搜索而非 ChatGPT —— 用户的目的是快速定位知识，而非让 AI 替自己思考。

**调整后的设计**：
- 移除 LLM 答案生成和评分功能
- 改为"综合关联度"排序：语义相似度（55%）+ 关键词匹配强度（45%）
- 每个文档展示**推荐理由**（如"标题匹配「微服务架构」，语义相似度 50%"）
- 每个文档展示**一句话摘要**
- 名称从「智能问答」→「知识推荐」→「知识查询」

**教训**：AI 驱动的产品容易陷入"技术炫技"的陷阱。团队内部知识库的核心价值是"精准检索"而非"智能对话"。

### 2.4 持续微调阶段（5/15 → 5/16）

这是投入时间最多的阶段，共计 17 次提交，全部围绕搜索结果质量：

**中文搜索优化**（难点）：
- 问题 1："微服务架构"搜不到文档 → 引入 bigram 匹配（标题中文二元组分词）
- 问题 2："Karpathy"返回 0 结果 → MEDIUM 条件遗漏 `en_hit`
- 问题 3："tsconfig.json"被"json"误匹配 → 英文词改为 `\b` 词边界 + 最少 3 字符
- 问题 4："Next.js"误匹配"labeler.js" → 同上修复
- 问题 5："PDF"返回 0 结果 → 新增 MEDIUM B（极短英文查询 + 正文命中）
- 问题 6："鸿蒙应用开发"返回 0 结果 → 标题为英文路径不含中文 → 新增 MEDIUM C（正文首 bigram 命中 + 语义 ≥ 50%）

**去重**：相同 `source+title` 只保留最高分文档。

**排序**：从分类展示（HIGH/推荐文档 + MEDIUM/可能相关）改为统一按综合关联度排列。

每一个问题都是从用户实际的搜索行为中发现的（"我搜了一下 XX，为什么没结果？"），然后逐一定位根因、修复、验证。

### 2.5 阶段性总结（5/16 → 5/18）

- 将「知识推荐」改名为「知识查询」
- 清理无效脚本（sync.sh）
- 补充 `.gitignore`
- 输出《离线版本部署方案》+ `.env.example` + `offline-deploy.sh`

**5/18 收尾工作**：
- 新增 MEDIUM B：极短英文查询（≤4字符）兜底，避免 "PDF"、"API" 等常见缩写搜不到
- 新增 MEDIUM C：中文查询标题无关键词时，用正文首 bigram 命中兜底（如"鸿蒙应用开发"）
- 去重逻辑：相同 `source+title` 只保留一条
- 补充《架构图》、《离线版本部署方案》文档
- 补充 `offline-deploy.sh` 一键部署脚本

## 三、技术架构

### 3.1 最终架构

```
用户浏览器 (:23100)
       │
  ┌────▼────────────────────┐
  │   WikiService (Flask)   │
  │  ┌──────────────────┐   │
  │  │ 知识查询  知识图谱 │   │
  │  │ 知识管理  数据导入 │   │
  │  └────────┬─────────┘   │
  │           │              │
  │  ┌────────▼─────────┐   │
  │  │ 检索引擎          │   │
  │  │ 语义搜索 + 关键词  │   │
  │  └────────┬─────────┘   │
  └───────────┼─────────────┘
              │ POST /api/embeddings
       ┌──────▼──────┐
       │   Ollama     │
       │ nomic-embed  │
       │  :11434      │
       └─────────────┘
```

### 3.2 搜索流程

```
查询 → Ollama 语义搜索（向量余弦相似度）
      ↓ 无结果
      BM25 关键词搜索（中文 bigram + 英文词边界）
      ↓
      综合关联度排序（语义 55% + 关键词 45%）
      ↓
      去重（source + title）→ 返回 Top 20
```

### 3.3 数据模型

```json
{
  "id": "chunk-xxx",
  "source": "github:owner/repo",
  "metadata": {"path": "README.md", "type": "file"},
  "text": "# 标题\n正文...",
  "created_at": "2026-05-15T10:00:00"
}
```

### 3.4 离线能力

- Ollama 本地运行 nomic-embed-text（274MB，一次下载永久离线）
- 所有数据本地 JSON 文件存储
- GitHub 文档首次在线拉取后持久化到本地
- 无需任何外部 API

## 四、设计、开发、测试、调整的方法论

### 4.1 我们做了什么

```
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  设计     │ →  │  开发     │ →  │  测试     │ →  │  调整     │
│ (ima +   │    │ (Claude  │    │ (用户    │    │ (根因    │
│  Claude) │    │  实现)   │    │  验证)   │    │  修复)   │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↑                                               │
      └───────────────────────────────────────────────┘
                    快速迭代闭环
```

**设计阶段**：
- ima 做需求分析和方案调研（横向对比 4+ 开源方案）
- Claude 参与架构讨论，输出技术方案文档
- 关键决策记录在文档中，包括"为什么选择 A 而不是 B"

**开发阶段**：
- Claude 按方案实现核心功能
- 每个功能点独立 commit，确保可追溯
- 保持代码简洁（~1400 行 Python），不过度抽象

**测试阶段**：
- 用户直接在浏览器中试用，输入真实查询
- 发现 bug 立即反馈，不等测试用例覆盖

**调整阶段**：
- 精确定位根因（如"为什么搜不到鸿蒙？→ 因为标题是英文路径 → text_hit 用了完整子串匹配 → 只检查了标题"）
- 小步修改、立即验证
- 每次修复只改一处逻辑，降低引入新 bug 的风险

### 4.2 对团队的启发

**一、AI 工具的正确用法**：
- AI 擅长"执行方案"，不擅长"判断方向"
- ima 的调研能力 + Claude 的编码能力 = 完整的开发流水线
- 但方案选型、方向调整、需求理解必须由人主导
- 我们在 6 天内完成通常需要 2-3 周的工作量，关键在于"人定方向、AI 执行"

**二、快速试错优于完美规划**：
- 初始的 WeKnora + Docker 方案虽然被弃用，但帮助确认了"离线单机"的方向
- 如果一开始就过度规划，可能还在纠结架构细节
- 建议：设定一个"最小可用版本"（如"能搜索就行"），48 小时内跑通，然后迭代

**三、搜索质量是"磨"出来的**：
- 没有一个搜索系统能一次设计到位
- 17 次微调提交都是针对具体场景的修复
- 建议：上线后预留 3-5 天的"搜索打磨期"，让用户实际使用并反馈

**四、文档先行，代码跟进**：
- 项目中有 6 篇设计/部署文档，都是在编码前完成的
- CLAUDE.md 作为 AI 助手的"上下文文件"，记录架构、部署约定、分支策略
- 建议：每个项目都应该有一份 CLAUDE.md，让 AI 更快理解项目

**五、从"AI 替你回答"到"AI 帮你检索"的认知转换**：
- 这是本项目最重要的产品洞察
- 很多 AI 知识库产品都在强调"对话式搜索"，但团队内部场景下，用户更需要的是精准定位文档
- 建议：在设计 AI 产品时，先问自己"用户到底要什么"，再问"AI 能做什么"
